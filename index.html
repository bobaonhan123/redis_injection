Blue Team: Defense and Verification

HMAC validation + safe parsing + NetworkPolicy

Now we switch to defenses. The goal is to block queue poisoning and reduce lateral movement if a pod is compromised.

Defense Layers

JSON + HMAC

Accept only valid UTF-8 JSON and verify the signature before processing.

Worker-secure

Remove pickle usage and log drop reasons (non-UTF8 / invalid JSON).

NetworkPolicy

Restrict internal traffic and reduce blast radius when a pod is compromised.

Three layers: strict data format, secure worker behavior, and network segmentation. Each layer cuts off a link in the chain.

Verification Checklist

  • ✓ Rejected malicious payload
  • ✓ Accepted valid signed payload
  • ✓ Malicious marker was not created
  • ✓ NetworkPolicy objects found

Script: python scripts/blue_team_verify.py --mode k8s ...

The verification script pushes both bad and good jobs. We expect the secure worker to accept only signed JSON, and to show policy objects.

Hardening Recommendations

  • Remove or lock SSRF endpoints in production
  • Ban unsafe serializers (pickle, yaml.load)
  • Disable automountServiceAccountToken when unused
  • Use admission policies to block risky pod specs
  • Alert on suspicious queue payload sizes
These are the most practical production controls. The top two remove the exploit path entirely; the rest reduce impact and improve detection.

Takeaways

Redis is not the execution target

Redis is the delivery channel. The bug lives in the consumer that deserializes untrusted data.

Defense-in-depth works

JSON + HMAC + NetworkPolicy reduces RCE risk and limits lateral movement.
